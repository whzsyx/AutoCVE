# DeepAudit 项目架构与漏洞发现流程说明

本文档基于对 `DeepAudit-3.0.0` 代码的阅读，总结项目架构、漏洞发现方式、规则/文件扫描匹配机制，以及从发现到确认的完整流程。

---

## 一、项目架构总览

DeepAudit 是一个**基于 Multi-Agent 协作的代码安全审计平台**，采用前后端分离与微服务式设计。

### 1.1 技术栈与目录结构

| 层级 | 技术 | 路径 |
|------|------|------|
| 前端 | React 18 + TypeScript + Vite | `frontend/` |
| 后端 | FastAPI + Python 3.11+ | `backend/` |
| 数据库 | PostgreSQL | 独立部署 / Docker |
| Agent 引擎 | LangGraph + 自研编排 | `backend/app/services/agent/` |
| 沙箱 | Docker | `docker/sandbox/` |

**后端核心目录：**

```
backend/
├── app/
│   ├── api/v1/endpoints/     # API：scan、agent_tasks、rules、projects 等
│   ├── core/                 # 配置、加密等
│   ├── db/                   # 数据库与会话
│   ├── models/               # ORM：Project、AuditTask、AuditRule、AgentTask、AgentFinding 等
│   ├── schemas/              # Pydantic 请求/响应模型
│   └── services/             # 业务核心
│       ├── scanner.py        # 仓库/文件扫描与 LLM 单文件分析（非 Agent）
│       ├── llm/              # LLM 调用、analyze_code / analyze_code_with_rules
│       ├── rag/               # RAG 索引与检索（代码语义检索）
│       └── agent/             # Multi-Agent 审计引擎
│           ├── agents/        # Orchestrator、Recon、Analysis、Verification
│           ├── core/          # 状态、注册表、图控制器、执行器
│           ├── tools/         # pattern_match、smart_scan、create_vulnerability_report、run_code 等
│           ├── knowledge/     # 漏洞知识库（CWE/类型）与框架知识
│           └── prompts/      # 系统提示词与多 Agent 规则
├── alembic/                  # 数据库迁移
└── tests/
```

### 1.2 两种审计模式

| 模式 | 入口 | 主要逻辑 | 结果存储 |
|------|------|----------|----------|
| **快速/经典扫描** | 创建审计任务 → `scan_repo_task` | `scanner.scan_repo_task`：拉取文件 → 逐文件调用 LLM（可选规则集+提示词） | `AuditTask` + `AuditIssue` |
| **Agent 深度审计** | 创建 Agent 任务 → 流式执行 | Orchestrator 调度 Recon → Analysis → Verification，子 Agent 用工具发现并报告漏洞 | `AgentTask` + `AgentFinding` |

## 二、漏洞是如何被发现的

漏洞发现依赖**多种检测手段**，在不同模式下组合方式不同。

### 2.1 快速/经典扫描模式（scanner）

- **扫描对象**：仓库或 ZIP 中的**文本源码文件**（按扩展名过滤，如 `.py/.js/.java` 等），排除 `node_modules/`、`__pycache__/`、`.git/` 等。
- **单文件分析**：对每个文件内容调用 LLM：
  - **无规则集**：`llm_service.analyze_code(code, language)`，使用内置系统提示词，要求 LLM 按安全/Bug/性能/风格等维度输出 JSON（issues + quality_score）。
  - **有规则集/提示词**：`llm_service.analyze_code_with_rules(code, language, rule_set_id=..., prompt_template_id=..., db_session=db)`，会从数据库加载**规则集（AuditRuleSet + AuditRule）**和**提示词模板（PromptTemplate）**，拼成自定义分析提示词再调用 LLM。
- **规则来源**：**数据库表** `audit_rule_sets`、`audit_rules`（规则集 + 单条规则：rule_code、name、description、category、severity、custom_prompt 等），**不是** 直接读 `rules/*.yml` 文件。任务创建时可传 `scan_config.rule_set_id` / `prompt_template_id`。
- **结果**：LLM 返回的每个 issue 被写成一条 `AuditIssue`（file_path、line_number、issue_type、severity、title、message、suggestion、code_snippet、ai_explanation 等）。

### 2.2 Agent 深度审计模式（Multi-Agent）

由 **Orchestrator** 决定步骤，依次调度 **Recon → Analysis → Verification**，漏洞由子 Agent 通过**工具**和 **LLM 推理** 发现，并最终通过 **create_vulnerability_report** 或 **Agent 返回的 findings** 汇总。

#### 2.2.1 信息收集（Recon Agent）

- 分析项目结构、技术栈、入口点、依赖等，为后续分析提供上下文；可产出 `initial_findings`、`high_risk_areas` 等，供 Orchestrator 转为“待分析/待验证”的发现。

#### 2.2.2 漏洞挖掘（Analysis Agent）

Analysis 的提示词要求**优先使用外部/内置扫描工具**，再结合 LLM 判断。漏洞来源包括：

1. **外部/内置扫描工具（按优先级使用）**
   - **semgrep_scan**：全语言静态规则（如 `p/security-audit` 或 auto）。
   - **bandit_scan**：Python 安全扫描。
   - **gitleaks_scan**：密钥泄露。
   - **safety_scan / npm_audit**：依赖漏洞。
   - **kunlun_scan**：Kunlun-M 深度审计（如 PHP/JavaScript）。
   - **smart_scan**：内置“智能批量扫描”，对目录/文件做模式+密钥+依赖等检查（内部用正则等）。
   - **pattern_match**：按**内置危险模式**做正则匹配（见下节）。

2. **RAG + 代码阅读**
   - **rag_query / security_search**：从已索引代码中做语义检索。
   - **read_file / list_files / search_code**：读文件、列目录、搜代码。

3. **LLM 综合判断**
   - 根据工具输出和代码内容，以 ReAct 形式决定下一步工具调用或直接给出 **Final Answer**，其中包含 `findings` 列表（每条为 dict：file_path、severity、vulnerability_type、description 等）。

Analysis 在 `run()` 中会收集：
- 各步中 LLM 输出的 **Final Answer 里的 findings**；
- 经 `_standardize_finding()` 标准化后，通过 `AgentResult(data={"findings": standardized_findings})` 返回给 Orchestrator。

#### 2.2.3 漏洞验证（Verification Agent）

- 对 Analysis（或 Recon）传来的可疑点进行**验证**：编写 PoC、在沙箱中 **run_code**、或使用 **sandbox_exec / sandbox_http** 等。
- 验证通过后，可调用 **create_vulnerability_report** 正式登记一条漏洞（写入内存中的报告列表）；也可在返回的 `AgentResult` 中带上 `findings`（含 is_verified、poc 等）。

#### 2.2.4 正式报告漏洞的唯一入口

- **create_vulnerability_report**（`reporting_tool.py`）：  
  参数包括 title、vulnerability_type、severity、description、file_path、line_start/end、code_snippet、poc、impact、recommendation、confidence、cwe_id、cvss_score 等。  
  - 会校验 severity、vulnerability_type、**且会校验 file_path 在 project_root 下是否真实存在**（防止幻觉）。  
  - 通过后写入 `CreateVulnerabilityReportTool._vulnerability_reports`（类级别列表），便于后续汇总。

## 三、漏洞是根据什么文件、什么规则做扫描/匹配的

### 3.1 扫描哪些文件

- **快速/经典扫描**（`scanner.py`）  
  - 由仓库类型（GitHub/GitLab/Gitea）或 SSH 拉取文件列表；或从 ZIP 解压得到文件。  
  - 只处理 **TEXT_EXTENSIONS** 中的扩展名（如 .js/.ts/.py/.java/.go/.php 等）且未被 **EXCLUDE_PATTERNS** 排除的文件；单文件大小受 `MAX_FILE_SIZE_BYTES` 限制。  
  - 若配置了 `scan_config.file_paths`，则**只分析这些指定路径**；否则按 `max_analyze_files` 截断数量。

- **Agent 模式**  
  - 项目根目录由任务上下文给出（例如克隆到本地或解压目录）。  
  - Recon 会列出/分析目录与入口；Analysis/Verification 通过 **list_files、read_file、pattern_match(scan_file=...)、smart_scan(target=...)** 等指定要扫描的路径或文件。  
  - 若创建 Agent 任务时传了 `target_files`，Orchestrator 会在初始消息中强调只审计这些文件。

即：**“根据什么文件”** = 仓库/ZIP 中的文本源码文件集合（或用户指定的 `file_paths`/`target_files`），再经扩展名、排除模式、大小、范围配置过滤。

### 3.2 规则/模式从哪里来（匹配逻辑）

- **数据库规则（仅用于快速/经典扫描的 LLM 分析）**  
  - **规则集**：`AuditRuleSet`（id、name、language、rule_type、severity_weights 等）。  
  - **单条规则**：`AuditRule`（rule_set_id、rule_code、name、description、category、severity、custom_prompt、fix_suggestion、reference_url 等）。  
  - 在 `analyze_code_with_rules()` 中：按 `rule_set_id` 查出规则集并 load 其下所有 enabled 的 `AuditRule`，转成 `rules` 列表（rule_code、name、description、category、severity、custom_prompt 等），与**提示词模板**一起拼进 LLM 的 system/user 内容，**由 LLM 根据规则描述做语义级判断**，而不是在后台做正则或 AST 匹配。

- **Agent 模式下的“规则”**  
  - **pattern_match**（`pattern_tool.py`）：内置 **PATTERNS** 字典，按漏洞类型（如 sql_injection、xss、command_injection、path_traversal、ssrf、deserialization、hardcoded_secret、weak_crypto）分语言（python、javascript、java、php、go 等）或 `_common`，每条是 **(正则, 说明)**。对传入的 `code` 或 `scan_file` 按行做 `re.search(pattern, line)`，命中即记一条 PatternMatch（pattern_type、file_path、line_number、matched_text、context、severity、description）。  
  - **smart_scan**（`smart_scan_tool.py`）：内部有 **QUICK_PATTERNS** 等（同样是按漏洞类型的正则列表），对选中文件做批量扫描，并可能结合目录/文件名风险模式（HIGH_RISK_PATTERNS）。  
  - **外部工具**：Semgrep 使用其自己的规则（如 p/security-audit）；Bandit/Gitleaks/Kunlun 等各有规则集，不在 DeepAudit 的 `rules/` 下配置。

- **rules/SeletItem.yml**  
  - 项目下存在 `rules/SeletItem.yml`，格式类似 Semgrep（id、language、files、rule.kind、rule.all、message、severity）。  
  - 在当前代码库的 **Python 后端** 中**未发现**对该 YAML 的加载或引用；它可能用于 Semgrep 外部规则、前端展示或预留扩展，**不参与当前 LLM 分析或 pattern_match/smart_scan 的匹配逻辑**。

总结：  
- **按文件**：仓库/ZIP 内文本文件 + 扩展名/排除/大小/可选 file_paths 或 target_files。  
- **按规则**：  
  - 经典扫描 = 数据库 `audit_rule_sets` + `audit_rules` + 提示词 → LLM 语义判断；  
  - Agent = 内置正则（pattern_tool + smart_scan）+ 外部工具（Semgrep/Bandit 等）+ LLM 对工具结果的解释与汇总。

## 四、从发现到确认的完整流程

### 4.1 快速/经典扫描

1. 用户创建审计任务（选项目、分支、可选规则集/提示词、file_paths 等）。  
2. **scan_repo_task**：拉取文件列表 → 对每个文件取内容 → 调用 `analyze_code` 或 `analyze_code_with_rules`。  
3. LLM 返回 JSON（issues 数组）。  
4. 每个 issue 写入 **AuditIssue**（task_id、file_path、line_number、issue_type、severity、title、message、suggestion、code_snippet、status=open 等）。  
5. 无“验证”环节；**“确认”**仅表示已写入库并可在前端标记状态。

### 4.2 Agent 深度审计（发现 → 汇总 → 确认 → 落库）

1. **任务创建与启动**  
   - 用户创建 Agent 任务（project_id、target_vulnerabilities、verification_level、target_files、exclude_patterns 等）。  
   - 后端克隆/解压项目到 `project_root`，建 RAG 索引（若需要），注册根 Agent（Orchestrator），开始流式执行。

2. **Orchestrator 循环（ReAct）**  
   - 每轮：LLM 输出 Thought / Action / Action Input。  
   - **dispatch_agent**：调用 Recon → 再根据结果 dispatch Analysis → 再根据结果 dispatch Verification（可选）。  
   - **summarize**：查看当前已汇总的 `_all_findings`。  
   - **finish**：结束审计，将 `_all_findings` 放入 `AgentResult.data["findings"]` 返回。

3. **子 Agent 结果如何变成“发现”**  
   - Recon：返回 project_structure、tech_stack、entry_points、initial_findings、high_risk_areas 等；Orchestrator 将 initial_findings 与 high_risk_areas 规范化为 dict 发现，并入 `_all_findings`。  
   - Analysis：通过工具（semgrep、pattern_match、smart_scan、rag_query、read_file 等）和 LLM Final Answer 得到 findings，标准化后通过 `AgentResult(data={"findings": standardized_findings})` 返回；Orchestrator 从 `result.data["findings"]` 取出，与现有 `_all_findings` 去重合并（按文件+行号/描述/类型）。  
   - Verification：对部分发现做 PoC/沙箱验证，可调用 **create_vulnerability_report** 写入正式报告，或在返回的 `AgentResult` 中带 findings（含 is_verified、poc）；Orchestrator 同样合并进 `_all_findings`，并保留已验证标记。

4. **结束与持久化**  
   - Orchestrator 执行 **finish** 后，返回 `AgentResult(success=True, data={"findings": self._all_findings, "summary": ...})`。  
   - 后端在 `agent_tasks` 的流程中拿到 `result.data["findings"]`，调用 **_save_findings(db, task_id, findings, project_root)**。

5. **_save_findings（确认与落库）**  
   - 对每条 finding：  
     - 规范化 severity、vulnerability_type、file_path、line_start/end、code_snippet、title、description、suggestion、confidence、is_verified、poc 等。  
     - **v2.1 文件路径校验**：若提供了 `project_root` 且 `file_path` 非空，则检查 `project_root/file_path` 是否真实存在；若不存在且也不是合法绝对路径，**丢弃该条**（视为幻觉），不写入库。  
   - 通过校验的发现写入 **AgentFinding** 表（task_id、vulnerability_type、severity、title、description、file_path、line_start/end、code_snippet、is_verified、ai_confidence、status、has_poc、poc_*、cvss_score 等）。  
   - 更新 AgentTask 的 findings_count、verified_count、*_count（按严重程度）等。

6. **对前端的“确认”含义**  
   - **已确认**：已通过 _save_findings 写入 DB，且通过文件存在性校验；  
   - **is_verified**：来自 Analysis/Verification 的验证结果或 create_vulnerability_report（该工具写入的报告默认 is_verified=True）；  
   - 用户还可在前端对某条 AgentFinding 做状态更新（如误报标记）。

### 4.3 流程小结（Agent 模式）

```
用户创建 Agent 任务
  → 准备 project_root、RAG、注册 Orchestrator
  → Orchestrator 循环：dispatch_agent(recon) → dispatch_agent(analysis) → [dispatch_agent(verification)]
  → 各子 Agent 通过工具（pattern_match / smart_scan / semgrep / create_vulnerability_report 等）和 LLM 产出 findings
  → 子 Agent 的 AgentResult.data["findings"] 被 Orchestrator 合并到 _all_findings（去重、合并验证状态）
  → Orchestrator 执行 finish，返回 data["findings"] = _all_findings
  → 后端 _save_findings：校验 file_path 存在性 → 写入 AgentFinding，更新 AgentTask 统计
  → 前端展示/导出报告、按严重程度与验证状态筛选
```

| `backend/app/services/agent/tools/rag_tool.py` | **rag_query**、**security_search**：代码语义检索，辅助定位可疑代码 |
